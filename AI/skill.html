Skill 技能文件详细说明 ​

本文整理了在 AI Agent 场景中，Skill 技能文件的用途、结构、编写规范和落地建议。

什么是 Skill ​

Skill 是给 Agent 的可复用能力模块，目标是让 Agent 在特定场景下遵循一致流程，而不是每次临时发挥。

常见用途：

  • 固化高频流程（例如：排查故障、发布前检查、代码评审）
  • 约束执行顺序（先检查，再修改，再验证）
  • 统一输出格式（结论模板、风险清单、回归项）
  • 降低多人协作的沟通成本

Skill 文件应包含什么 ​

一个实用的 Skill，至少应包含以下部分：

  1. 触发条件：什么情况下必须使用这个 Skill
  2. 目标与边界：要解决什么问题，不解决什么问题
  3. 执行步骤：按顺序执行的操作清单
  4. 输入输出约定：需要哪些输入、产出什么结果
  5. 验收标准：什么叫“完成”
  6. 失败回退策略：执行失败时如何处理

推荐目录结构（含 script、reference） ​

建议一个 Skill 使用独立目录，目录名语义化，例如 debug-api-timeout/。

text
skills/
  debug-api-timeout/
    SKILL.md
    scripts/
      collect-logs.sh
      check-metrics.sh
    references/
      runbook.md
      faq.md
    examples/
      input.json
      output.md

各目录职责：

  • SKILL.md：主入口，定义触发条件、步骤、输出规范
  • scripts/：可执行脚本（自动化采集、检查、格式化等）
  • references/：参考资料（SOP、架构图说明、术语、FAQ）
  • examples/：输入输出示例，帮助使用者快速对齐格式

命名建议：

  • 目录与文件统一小写短横线：如 collect-logs.sh
  • reference 建议统一写成 references（复数更语义化）
  • 脚本按动作命名：check-*、collect-*、generate-*

推荐结构（SKILL.md 模板） ​

md
# <Skill 名称>

## 适用场景
- ...

## 不适用场景
- ...

## 前置条件
- ...

## 执行流程
1. ...
2. ...
3. ...

## 输出规范
- ...

## 验证清单
- [ ] ...
- [ ] ...

编写原则 ​

1) 先约束，再赋能 ​

先写清楚“不能做什么”，再写“怎么做”。
例如：禁止跳过验证、禁止直接删库删表、禁止未确认就发布。

2) 步骤必须可执行 ​

避免写“适当处理”“必要时优化”这类模糊描述。
每一步都应可落地，最好带命令、路径、判断条件。

3) 输出要可检查 ​

结果不是“已完成”四个字，而是：

  • 改了哪些文件
  • 运行了哪些验证
  • 剩余风险是什么

4) 兼容协作与自动化 ​

尽量使用标准化格式，便于后续脚本、CI、review 直接消费。

质量检查清单 ​

提交 Skill 前，可快速自检：

  • [ ] 触发条件是否明确（避免“万物皆可用”）
  • [ ] 执行步骤是否有顺序且可复现
  • [ ] 输出结构是否统一
  • [ ] 是否包含异常分支与回退策略
  • [ ] 是否和现有流程冲突

示例：排查类 Skill（简化版） ​

md
## 适用场景
- 用户反馈线上报错、接口超时、页面白屏

## 执行流程
1. 收集报错上下文（时间、环境、版本、复现路径）
2. 定位日志与监控信号（error rate、latency、关键 trace）
3. 判断影响范围（单用户 / 局部 / 全量）
4. 给出临时止血方案与根因修复方案
5. 补充验证与复盘结论

与普通文档的区别 ​

  • 普通文档偏“说明”
  • Skill 更偏“执行协议”

如果一篇文档无法直接指导 Agent 执行任务，它更像知识文档，而不是 Skill。

维护建议 ​

  • 小步迭代：每次只改一个流程点
  • 版本可追踪：每次变更记录“为什么改”
  • 定期清理：移除失效步骤和过时命令

目录级最佳实践 ​

scripts ​

  • 每个脚本只做一件事，避免“巨型脚本”
  • 脚本开头写用途、输入参数和退出码说明
  • 对危险操作加显式确认（例如删除、覆盖、发布）

references ​

  • 只放稳定知识，不放临时结论
  • 每篇文档开头写“适用范围”和“最后更新时间”
  • 外部链接建议附一行摘要，避免失效后无法理解

examples ​

  • 至少提供 1 组“最小可运行”样例
  • 同时提供“成功输出”和“失败输出”示例
  • 示例里的路径、命令和字段要与 SKILL.md 保持一致

Skill 发展历史（背景脉络） ​

你提到的方向是对的：Skill 的实践最早在 Agent 编程社区中被系统化，典型代表就是 Claude Code 这类工作流。它的核心思想是把“提示词”升级成“可复用流程单元”。

早期阶段：Prompt 模板时代 ​

  • 主要靠一段长 prompt 重复完成任务
  • 问题是稳定性差，换人后效果波动明显
  • 很难沉淀为团队资产

中期阶段：Claude Code 等工具推动 Skill 化 ​

  • 社区开始把高频任务封装成独立 SKILL.md
  • 通过“触发条件 + 步骤清单 + 输出规范”提高一致性
  • 从“会聊天的助手”转向“可执行的流程代理”

近期阶段：Skill 工程化 ​

  • 单文件 Skill 扩展为目录化组织（scripts/、references/、examples/）
  • 与代码仓库、CI、评审流程深度结合
  • 形成团队级知识与执行标准

为什么 Claude Code 场景里 Skill 很重要 ​

  • 任务可复现：同样输入，流程和输出更稳定
  • 经验可传承：资深同学的方法能沉淀下来
  • 协作可对齐：新人按 Skill 就能跑通流程
  • 风险可控制：关键步骤不会被遗漏

你可以这样理解 Skill 的历史价值 ​

Skill 不是“另一种文档”，而是 Agent 时代的软件工程实践：
把个人技巧变成团队协议，把临时发挥变成可重复执行的流程。

什么时候封装成 Skill ​

满足下面任意 2 条，就建议封装：

  • 同一类任务一周内重复 3 次以上
  • 每次都要手动复制一段长提示词或检查清单
  • 新同学执行时容易漏步骤、结果不一致
  • 任务有固定输入输出格式（如排障报告、发布清单）
  • 任务涉及高风险操作，需要强约束流程

不建议封装的场景：

  • 一次性需求，后续不会复用
  • 需求仍在频繁变化，流程还没稳定
  • 只是简单知识点，不是执行流程

推荐 5 个常用 Skill ​

括号里的英文名（如 tdd-workflow、pre-release-check）是自建 Skill 时推荐的文件夹 / 仓库命名，方便团队统一；不是各「Skill 市场」上的商品标题，所以在市场里按这类内部命名去搜，经常搜不到，属于正常现象。

市场里请用下面「参考搜索」里的自然语言关键词（或自己改词再搜）。

  1. 系统化排障
    用于报错、超时、线上故障；固定“收集上下文 -> 定位 -> 验证 -> 结论”。
    参考搜索：AgentSkills troubleshooting ｜ AgentSkills incident ｜ AGSkills on-call

  2. 测试优先开发（建议自建名：tdd-workflow）
    用于功能开发或缺陷修复；固定“先写失败用例 -> 实现 -> 回归”。
    参考搜索：AgentSkills tdd ｜ AGSkills test-driven

  3. 代码评审（建议自建名：code-review-checklist）
    用于提测或合并前；重点检查风险、边界、回归项、可维护性。
    参考搜索：AgentSkills code review ｜ AGSkills pull request

  4. 发布前检查（建议自建名：pre-release-check）
    用于上线前；覆盖配置、依赖、监控、回滚预案、验收清单。
    参考搜索：AgentSkills release checklist ｜ AGSkills deploy

  5. 文档沉淀
    用于任务收尾；统一生成变更说明、使用指南、已知问题。
    参考搜索：AgentSkills release notes ｜ AgentSkills changelog ｜ AGSkills documentation

Skill 市场网站（可直接找现成 Skill） ​

  • AgentSkills：聚合型 Skill 市场，支持按平台/分类检索
  • AGSkills：开源 Skill 目录，适合快速浏览热门模板
  • SkillsMK：社区聚合站，可按关键词找 SKILL.md 资源
  • SkillsAuth：强调安全扫描与校验的 Skill 市场
  • ClawHub：Skills / Plugins 社区，支持按下载量、分类等浏览与发布
  • SkillHub：面向中国用户的 Skills 社区
  • Cursor Marketplace：Cursor 官方市场（插件中可包含 skills）

建议：优先选有最近更新时间、示例完整、可见源码地址的 Skill。
